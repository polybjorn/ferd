# Atlas

A self-hosted Leaflet map for plotting travel places and GPX trails. Single-file static frontend, optional Python API for in-browser editing. No build step, no Docker.

## Features

- Unified world map with clustered place pins and GPX trail polylines.
- Filter panel: per-category place toggles, visited/want chips, completed/planned trail chips.
- Browse views: places grouped by category or country; trails grouped by region.
- Trail detail view with elevation profile and route stats.
- Search across both places and trails.
- Self-hosted accounts with first-run registration and an in-app "add place" / "upload GPX" flow. Optional; the site works read-only with no API.

## Quickstart

### View only

Serve the directory with any static web server. Edit `places.json` and the `gpx/` tree by hand (run `./gpx-manifest.sh` after GPX changes to regenerate `routes.json`).

```sh
python3 -m http.server 8090
# http://localhost:8090/
```

### View + edit, single process (for trying it out)

```sh
cp tools/config.example.json tools/config.json
cp site-config.example.json site-config.json

ATLAS_DATA_DIR="$PWD" \
ATLAS_STATIC_DIR="$PWD" \
ATLAS_MANIFEST_CMD="$PWD/gpx-manifest.sh" \
ATLAS_SECURE_COOKIES=false \
ATLAS_BIND=127.0.0.1:8090 \
  python3 tools/api.py
```

Open `http://127.0.0.1:8090/`, click "Sign in", register the first account, then use the "+" button to add places or trails.

## Configuration

Two config files, both optional, both gitignored:

- **`site-config.json`** (read by the browser): brand, default map view, category labels, API base path. See `site-config.example.json` for the schema.
- **`tools/config.json`** (read by the API server): listen address, data directory, manifest command, password / cookie / body-size policies, optional pre-seeded operator credentials. See `tools/config.example.json`.

Every key in `tools/config.json` is also overridable via an `ATLAS_<UPPER>` environment variable.

## Editing data

Three paths reach the same files, in parallel:

1. **In the browser**: sign in, click "+", fill the form, submit. The API validates, atomically writes `places.json` (or the GPX file under `gpx/<region>/`), strips PII from uploaded GPX (`<time>`, `<author>`, `creator=`), and re-runs the manifest.
2. **In a text editor**: edit `places.json` or drop GPX files into `gpx/<region>/`, then run `./gpx-manifest.sh`. Atlas reloads them on the next page load.
3. **From a script**: post JSON to `/api/places` or raw GPX to `/api/gpx?region=X&name=Y` with a valid session cookie.

Atomic writes plus a file lock keep same-host concurrent writes safe. Cross-device concurrency (e.g. text-editor edits on one machine while submitting forms from another, with Syncthing in between) is bounded by "use one writer at a time" - if two machines write the same file simultaneously you'll get a Syncthing conflict sibling.

## Security

See the "Security notes" section in [deploy/README.md](deploy/README.md). Short version:

- Don't expose Atlas to the open internet between deploying and finishing first-run registration, or pre-seed the operator account so registration is already closed.
- Passwords are PBKDF2-SHA256, 600k iterations.
- Sessions are cookie-based (`HttpOnly`, `SameSite=Lax`, `Secure` on by default).
- Login is rate-limited (10 failed attempts per IP per 15 minutes) and timing-equalized so usernames don't leak via response time.
- The SQLite users + sessions DB is created mode 0600.
