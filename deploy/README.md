# Deploying Atlas

Atlas ships as a static site plus an optional Python API. You can run it three ways:

- **View only**: drop the static files on any web host.
- **View + edit, on your own server**: run the Python API alongside, behind nginx.
- **View + edit, just for yourself**: run `python3 tools/api.py` directly with `static_dir` set so one process serves both static and API.

Pick the section that matches.

## 1. View only (no editing)

Atlas is a single HTML file plus a few static assets. Any web server works.

```sh
# Whatever static host you like:
rsync -a index.html favicon.svg site-config.example.json places.json gpx/ routes.json user@host:/var/www/atlas/
```

Things to know:
- `places.json` and `routes.json` are the data. `routes.json` is generated by `gpx-manifest.sh` from the contents of `gpx/`.
- Rename `site-config.example.json` to `site-config.json` and edit the brand, default view, and category labels for your install.
- Browsing works without the API. The "Sign in" button will just be a dead end.

## 2. View + edit, with the API and nginx

You'll need:
- A Linux box (or anything that runs Python 3.9+, systemd assumed below).
- A reverse proxy (this guide uses nginx).
- TLS (Let's Encrypt is easiest; or run plain HTTP behind a VPN like Tailscale).

### 2a. Files

Lay out the install at `/srv/atlas`:

```
/srv/atlas/
  index.html
  favicon.svg
  site-config.json          # your copy of site-config.example.json
  places.json               # gitignored data
  gpx/                      # gitignored data
  routes.json               # generated
  gpx-manifest.sh
  tools/
    api.py
    config.json             # your copy of tools/config.example.json
```

### 2b. API config

Edit `tools/config.json`:

```json
{
  "bind": "127.0.0.1:8091",
  "db_path": "/srv/atlas/tools/atlas.db",
  "data_dir": "/srv/atlas",
  "manifest_cmd": "/srv/atlas/gpx-manifest.sh",
  "secure_cookies": true,
  "max_body_bytes": 1048576,
  "idle_exit_seconds": 300,
  "static_dir": null
}
```

If you want to skip the browser registration step and pre-seed the operator account, also set:

```json
"initial_user": "you",
"initial_password": "CHANGE-ME-MIN-12-CHARS"
```

The account is created on first start; on later starts these fields are ignored.

### 2c. systemd units (socket-activated, zero-idle)

Copy the unit files into place:

```sh
sudo cp deploy/atlas-api.socket /etc/systemd/system/
sudo cp deploy/atlas-api.service /etc/systemd/system/
sudo useradd --system --home /srv/atlas --shell /usr/sbin/nologin atlas
sudo chown -R atlas:atlas /srv/atlas
sudo systemctl daemon-reload
sudo systemctl enable --now atlas-api.socket
```

How this works:
- `atlas-api.socket` holds the port at zero cost.
- The first request triggers `atlas-api.service` to start.
- After `idle_exit_seconds` of no activity, the Python process exits cleanly. The socket stays held. Next request wakes it again.

If you'd rather run it always-on, set `idle_exit_seconds` to 0 and `systemctl enable --now atlas-api.service` directly (skip the socket).

### 2d. nginx

Copy and edit `deploy/nginx.example.conf` to your nginx config dir. Replace `atlas.example.com`, the cert paths, and `/srv/atlas`.

Reload:

```sh
sudo nginx -t && sudo systemctl reload nginx
```

### 2e. First sign-in

Open `https://atlas.example.com/`. The first registration creates the operator account and closes registration. From the menu in the nav, "Settings" lets you change your password or re-open registration if you want to invite someone.

### 2f. macOS

There's a sample `deploy/atlas-api.plist`. macOS launchd doesn't have socket activation in the same shape as systemd; the plist just keeps the process running. Use `idle_exit_seconds: 0` so the process doesn't quietly exit on idle.

## 3. View + edit, just for yourself (one-process dev mode)

If you just want it running on your laptop, skip nginx entirely. The API can also serve the static files:

```sh
ATLAS_DATA_DIR="$PWD" \
ATLAS_STATIC_DIR="$PWD" \
ATLAS_MANIFEST_CMD="$PWD/gpx-manifest.sh" \
ATLAS_SECURE_COOKIES=false \
ATLAS_BIND=127.0.0.1:8090 \
  python3 tools/api.py
```

Open `http://127.0.0.1:8090/`. `secure_cookies=false` is required for plain HTTP. Don't expose this to the network.

## Security notes

- The API ships with no auth out of the box; registration is open until the first user signs up, then it closes. **Do not expose Atlas to the open internet between deploying and finishing first-run registration.** Run the first registration before opening the firewall, or deploy behind a VPN, or set `initial_user`/`initial_password` so the operator account is created on start and registration is already closed when the first request arrives.
- The SQLite DB at `tools/atlas.db` holds password hashes and active session tokens. Permissions are set to 0600 automatically. Back it up alongside your data.
- Atlas writes to `places.json` and `gpx/`. Both are atomic (tmp file + rename + fsync) and protected by a file lock against concurrent same-host writes. If you also edit these files in a text editor across devices via Syncthing or similar, be aware that simultaneous writes from two machines can produce conflict files; pick one canonical writer.

## Backups

The data is in `places.json` and `gpx/`; the users + sessions are in `tools/atlas.db`. Three files / paths total. Any backup tool (rsync, restic, borg) handles them; just make sure the API isn't mid-write when you snapshot the DB.
